## 0. 全体サマリー

### この事業が狙う空白

日本には「経営がしんどい」という初期の段階で、**匿名・無料・ジャッジなし**で話せる場所がない。弁護士や税理士は「まだそこまでじゃない」、商工会は「知り合いに見られたくない」、銀行は「追加融資を断られそう」、金融機関の相談窓口は「出向いてまで行く元気がない」。

その結果、経営者は孤立したまま意思決定が遅れ、傷が深くなってから専門家のドアを叩く。このサービスは**「専門家に相談する前の、最初の一歩」**を担う。

### 事業の本質

```
経営者の孤立 → 構造化された相談投稿 → 経験者・支援者からの回答 → 専門家/支援機関へのつなぎ
```

- 運営は**「場の提供者」**であり、法的判断・税務判断・代理行為は行わない
- **相談者（悩みを持つ経営者）** と **回答者（経験者・支援者）** のロールを明確に分ける
- 相談は**フェーズ別（資金繰り・税金・係争・再生・破産・清算）**に構造化し、適切な回答者にマッチさせる
- Yahoo!ファイナンス掲示板のような**スレッド型**のやりとりで、深いコミュニケーションを実現

### v2 → v3 での主要変更点

| 項目 | v2 | v3 |
|------|-----|-----|
| ユーザーロール | 一律「会員」 | **相談者** / **回答者** に分離 |
| 投稿形式 | 自由テキスト + カテゴリ選択 | **フェーズ選択 + 困りごとプルダウン + テキスト** の構造化入力 |
| 返信形式 | フラットなコメント | **スレッド形式**（返信に返信ができる） |
| 通知 | コメント通知のみ | **双方向通知**（相談者←→回答者） |
| 閲覧数 | DB管理のみ | **UI上に閲覧件数を表示** |

### 成立条件（先取りサマリー）

| 条件 | 内容 |
|------|------|
| 法的安全線の維持 | 「場の提供」に徹し、個別判断・代理・紹介手数料を取らない |
| 信頼性の構築 | 運営の透明性、専門家監修の明示、断定的アドバイスの抑止 |
| 安全なコミュニティ運営 | モデレーション体制、通報設計、危機対応フロー |
| 最初の100人の熱量 | 相談者50人 + 回答者50人の初期バランス確保 |
| 持続的マネタイズ | 信頼性を損なわない収益モデル（掲載課金・寄付・協賛） |

## 1. この事業の社会的意義

### なぜ今このサービスが必要か

#### マクロ背景

- 中小企業の**休廃業・解散件数**は年間5〜6万件超（帝国データバンク）
- コロナ禍のゼロゼロ融資（無利子・無担保融資）の返済が2023〜2025年に集中し、**資金繰り悪化が急増**
- 物価高・人件費上昇・人手不足が重なり、**黒字倒産・自主廃業の増加**が続く
- 一方で「倒産件数の増加」はまだ社会的なタブーとして語られにくい

#### 既存の相談先との比較

| 相談先 | 強み | 弱み（このサービスが埋める部分） |
|--------|------|-------------------------------|
| 弁護士 | 法的判断、交渉代理 | 費用・敷居・「まだそこまでじゃない」感 |
| 税理士 | 税務・帳簿 | 月次顧問が主、経営苦境の相談は範囲外感 |
| 商工会・商工会議所 | 補助金案内、経営相談 | 知り合いに見られる、形式的、地域差 |
| よろず支援拠点 | 無料・中立 | 予約制・対面・情報が拡散しない |
| 金融機関 | 融資・リスケ | 貸手の立場・追加融資を断られる恐れ |
| 事業再生ADR | 中立的再生支援 | 中規模以上向け・費用・専門性が高すぎる |
| 自治体窓口 | 無料・公的 | 縦割り・担当者の質差・解決に時間がかかる |
| **このサービス** | **匿名・24時間・ジャッジなし・経験者の回答** | 個別法律/税務判断はできない |

#### このサービスが埋める「空白」

1. **時間的空白**: 相談窓口が開いていない深夜・早朝に不安が増幅する
2. **感情的空白**: 「恥ずかしい」「弱音を吐けない」という孤立感
3. **情報的空白**: 「再生した人の実体験」が公には出てこない
4. **前段空白**: 「専門家に相談すべきか」すら判断できない段階
5. **構造的空白**: 自分の悩みが何に分類されるかすら分からない → **フェーズ別構造化で解消**

## 3. ユーザーロール設計（🆕 v3）

### ロール定義

| ロール | 定義 | 登録時の選択 |
|--------|------|-------------|
| 相談者 | 経営の悩みを相談する側。投稿を作成し、回答を受け取る | 「相談したい」を選択 |
| 回答者 | 経験や知見をもとに回答する側。相談に返信する | 「回答したい（経験者・支援者）」を選択 |
| 両方 | 相談も回答もする | 「両方」を選択 |
| モデレーター | 運営が任命した通報対応担当 | 運営から付与 |
| 管理者（運営） | 全権限を持つ運営スタッフ | 初期設定 |

### ロール別できること

| アクション | 未ログイン | 相談者 | 回答者 | 両方 | モデレーター | 管理者 |
|-----------|-----------|--------|--------|------|------------|--------|
| 相談一覧閲覧 | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ |
| 相談詳細閲覧 | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ |
| 相談投稿（新規相談） | ❌ | ✅ | ❌ | ✅ | ❌ | ✅ |
| 回答投稿（相談への返信） | ❌ | ❌ | ✅ | ✅ | ✅ | ✅ |
| 返信への返信（スレッド内） | ❌ | ✅（自分の相談内のみ） | ✅ | ✅ | ✅ | ✅ |
| 共感ボタン | ❌ | ✅ | ✅ | ✅ | ✅ | ✅ |
| 通報送信 | ❌ | ✅ | ✅ | ✅ | ✅ | ✅ |
| 自分の投稿編集 | ❌ | ✅ | ✅ | ✅ | ✅ | ✅ |
| 自分の投稿削除 | ❌ | ✅ | ✅ | ✅ | ✅ | ✅ |
| 他人の投稿削除 | ❌ | ❌ | ❌ | ❌ | ✅ | ✅ |
| 通報一覧確認 | ❌ | ❌ | ❌ | ❌ | ✅ | ✅ |
| アカウント停止 | ❌ | ❌ | ❌ | ❌ | ❌ | ✅ |
| 退会 | ❌ | ✅ | ✅ | ✅ | ❌ | ✅ |

### ロール選択の設計意図

- **相談者**は安心して悩みを打ち明けられる。回答者からの「上からのアドバイス」ではなく「寄り添い」を期待
- **回答者**は自分の経験・知見を活かせる。回答者のプロフィールに「経験フェーズ」を表示し信頼性向上
- **ロール切替は後から可能**（マイページ設定から「両方」に変更可）
- 回答者は、回答の前に**「個人的な経験・見解としてお伝えします」**チェックを必須とする（非弁リスク対策）

## 5. スレッド型返信設計（🆕 v3）

### スレッド構造

Yahoo!ファイナンス掲示板のように、**相談（スレッド）→ 回答 → 回答への返信**の3階層で構成する。

```
相談（スレッド元） ← 相談者が投稿
├── 回答 #1 ← 回答者が投稿
│   ├── 返信 1-1 ← 相談者 or 回答者
│   ├── 返信 1-2 ← 相談者 or 回答者
│   └── 返信 1-3
├── 回答 #2
│   ├── 返信 2-1
│   └── 返信 2-2
└── 回答 #3
    └── 返信 3-1
```

### 階層制限

- **最大3階層**（相談 → 回答 → 返信）。返信の返信はフラットに同階層に表示する
- これ以上ネストさせると可読性が落ちる
- 設計意図: 質問→回答のコンテキストを維持しつつ、深いスレッド化による混乱を防ぐ

### スレッド表示UI仕様

```

│ 💰 資金繰り > 銀行借入の返済が厳しい / 運転資金が足りない │
│                                                     │
│ 📌 このサービスは経営者が匿名で話せる相談の場です。      │
│    投稿内容は個人の経験・意見であり、専門的判断では       │
```

### スレッド内のルール

- **相談者**は自分の相談スレッド内ですべての回答に返信できる
- **回答者**は相談に対する回答を投稿でき、自分の回答内のスレッドで返信のやりとりもできる
- 回答投稿時は**「個人的な経験・見解としてお伝えします」チェック必須**
- 各回答・返信に共感ボタン・通報ボタンを設置

## 6. 双方向通知設計（🆕 v3）

### 通知トリガー一覧

| トリガー | 通知先 | 通知手段 | MVP対応 |
|---------|--------|---------|---------|
| 相談に回答がついた | 相談者 | メール | ✅ MVP |
| 回答に返信がついた | 回答者 | メール | ✅ MVP |
| 自分の返信に返信がついた | 返信元ユーザー | メール | ✅ MVP |
| 投稿に共感がついた | 投稿者 | メール（設定ON時のみ） | 後回し |
| 通報対応が完了した | 通報者 | メール | ✅ MVP |
| アカウント警告 | 対象ユーザー | メール | ✅ MVP |

### 通知メール設計

**件名例:**

| トリガー | 件名 |
|---------|------|
| 回答がついた | 「あなたの相談に回答がありました」 |
| 返信がついた | 「あなたの回答に返信がありました」 |
| 返信への返信 | 「スレッドに新しい返信がありました」 |

**メール本文方針:**
- **投稿本文はメールに含めない**（匿名性保護）
- 「回答がありました。サイトで確認してください。」＋ リンクのみ
- 通知メールの解除リンクを必ず設置
- 通知頻度が高すぎないよう、同一スレッドからの通知は**1時間以内にまとめて1通**（ダイジェスト方式）

### 通知設定

| 設定項目 | デフォルト | 変更可能 |
|---------|-----------|---------|
| 回答通知（相談者） | ON | ✅ |
| 返信通知（回答者） | ON | ✅ |
| 共感通知 | OFF | ✅ |
| メール通知一括OFF | OFF | ✅ |

## 9. 画面フロー（MVP v3）

### フロー1: 新規登録（相談者）

```
LP（トップページ）
    ↓ 「相談してみる」ボタン
会員登録画面
    ↓ ロール選択：「相談したい」
    ↓ メールアドレス・パスワード・ニックネーム・利用規約同意
認証メール送信完了画面
    ↓ メールのリンクをクリック
認証完了 → 相談一覧画面へ自動遷移
    ↓
初回のみ：サービス説明モーダル
（「ここは経験者同士の相談の場です。法的判断は専門家へ」）
```

### フロー2: 新規登録（回答者）

```
LP（トップページ）
    ↓ 「経験を活かして回答する」ボタン
会員登録画面
    ↓ ロール選択：「回答したい（経験者・支援者）」
    ↓ メールアドレス・パスワード・ニックネーム・利用規約同意
回答者プロフィール入力画面
    ↓ 経験フェーズ選択（複数可）、簡単な自己紹介（任意）
認証メール送信完了画面
    ↓ メールのリンクをクリック
認証完了 → 相談一覧画面へ自動遷移
    ↓
初回のみ：回答ガイドモーダル
（「個人的な経験・見解として回答してください。個別の法的判断は控えてください」）
```

### フロー3: 相談投稿

```
相談一覧画面
    ↓ 「相談する」ボタン
相談投稿画面
    ↓ Step 1: フェーズ選択（アイコンタイル）
    ↓ Step 2: 困りごとプルダウン（複数選択可）
    ↓ Step 3: タイトル入力
    ↓ Step 4: 本文入力
    ↓ 注意事項表示（「個人名・社名を含めないでください」）
プレビュー確認 → 投稿
    ↓
投稿完了（「相談が公開されました。回答を待ちましょう。」）
    ↓
相談詳細画面へ
```

### フロー4: 回答投稿

```
相談詳細画面（回答者がアクセス）
    ↓ 「回答する」ボタン
回答入力フォーム（相談詳細画面下部に展開）
    ↓ ☑「個人的な経験・見解としてお伝えします」チェック（必須）
    ↓ 回答テキスト入力
回答送信
    ↓
相談詳細画面に回答が表示される
    ↓
相談者にメール通知:「あなたの相談に回答がありました」
```

### フロー5: 返信（スレッド内）

```
相談詳細画面 → 特定の回答
    ↓ 「返信する」ボタン
返信入力フォーム（回答の下に展開）
    ↓ 返信テキスト入力
返信送信
    ↓
スレッド内に返信が表示される
    ↓
回答者にメール通知:「あなたの回答に返信がありました」
（逆の場合は相談者に通知）
```

### フロー6: 通報

```
相談詳細 / 回答 / 返信
    ↓ 「通報する」リンク
通報画面 → 理由選択 + 任意コメント
送信 → 「24時間以内に対応します。」
    ↓
運営に通報通知メール → 管理画面で対応
```

### フロー7: ロール変更

```
マイページ
    ↓ 「ロール設定」
現在のロール表示
    ↓ 「両方（相談も回答も）」に変更ボタン
確認ダイアログ → 変更完了
（回答者→両方の場合は追加の注意事項表示なし）
（相談者→両方の場合は回答ガイドモーダル表示）
```

## 21. バックアップ・障害復旧方針

### バックアップ方針

| 対象 | 方法 | 頻度 | 保持期間 |
|------|------|------|---------|

| ソースコード | GitHub | 常時（Git） | 無期限 |
| 環境変数 | Cloudflare Workers Secrets + wrangler.toml | 常時 | — |

### 障害対応フロー

```
障害検知（Sentry / Cloudflare Status / ユーザー報告）
    ↓
├─ Cloudflare Workers 障害 → Cloudflare Status 確認。復旧待ち
├─ Supabase 障害 → Supabase Status 確認。復旧待ち
├─ アプリバグ → wrangler rollback で即時ロールバック
└─ データ破損 → Supabase バックアップから復元
    ↓
復旧確認 → ユーザー告知
```

> **Cloudflare Workers のロールバック**: `wrangler rollback` コマンドで前のバージョンに即座に戻せる。Vercel の Instant Rollback と同等の機能。

## 24. 実装方針

### 推奨技術スタック（Cloudflare Workers 中心）

| レイヤー | 技術 | 理由 |
|---------|------|------|
| フロントエンド | Next.js 15 (App Router) | SSR/SSG対応、Reactエコシステム |
| CF適応 | @opennextjs/cloudflare | Next.jsをCloudflare Workers上で動かす公式推奨アダプター |
| UIライブラリ | Tailwind CSS + shadcn/ui | 高速UI構築、カスタマイズ性 |
| バックエンド/DB | Supabase (PostgreSQL + Auth) | 認証・RLS・Realtime対応 |
| ホスティング | Cloudflare Workers | エッジ配信（世界330+拠点）、無料枠が大きい |
| 静的アセット | Cloudflare Workers Static Assets | ビルド済みHTML/CSS/JS/画像の配信 |
| KVストア | Cloudflare KV | レート制限カウンター、セッションキャッシュ |
| メール | Resend | シンプル、月10,000通無料 |
| 決済（将来） | Stripe Payment Links | 寄付用。MVP後 |
| 監視 | Sentry（Cloudflare Workers対応SDK） | エラートラッキング |
| 分析 | GA4 + Cloudflare Web Analytics | 無料・プライバシーフレンドリー |
| DNS/CDN/WAF | Cloudflare（統合） | DNS・CDN・WAF・DDoS保護を一元管理 |
| CI/CD | Cloudflare Workers（wrangler deploy） | `wrangler` CLIでデプロイ |

### Cloudflare Workers を選択する理由

1. **コスト**: 無料枠が1日10万リクエスト（Workers Free）。有料プランも$5/月で1,000万リクエスト/月
2. **パフォーマンス**: エッジネットワーク330+拠点。日本国内ユーザーへの配信が高速
3. **統合**: DNS・CDN・WAF・Workers・KV がすべてCloudflareダッシュボードで一元管理
4. **スケーラビリティ**: 自動スケール。サーバー管理不要
5. **セキュリティ**: DDoS保護、WAFが標準装備

### Edge Runtime での注意事項

| 制約 | 影響 | 対策 |
|------|------|------|
| メモリ上限 128MB | 大きなJSON処理に制限 | このMVPはテキスト中心なので問題なし |
| Node.js API の一部が使えない | `fs`, `net`, `child_process` 等 | Supabase JS クライアントはEdge互換 |
| `@supabase/ssr` の互換性 | 一部 `stream` 依存エラーの報告あり | `@supabase/supabase-js` をクライアント中心に使用。SSR認証は Cloudflare Workers の `request.headers` からCookie取得で対応 |
| Cold start | Workers はV8 isolateで起動が速い（< 5ms） | Vercel Edge Functions より高速 |

## 25. 結論

### この事業が成立する条件

1. **法的安全線の徹底**: 「場の提供者」であり続け、個別判断・紹介料・代理を行わない
2. **信頼性の可視化**: 運営の透明性、専門家アドバイザーの明示、回答者の「個人的見解」チェック
3. **相談と回答のバランス**: 最初の50相談に対して回答がつくこと。回答者の初期確保が最重要
4. **創業者自身のストーリー**: Noriさんのトラーナ民事再生の経験が最初の回答者・最初の相談者としてのロールモデルになる
5. **持続可能な収益**: 信頼を損なわない掲載課金・支援会員モデルへの移行

### 最大のリスク

| リスク | 対策 |
|--------|------|
| 回答者が集まらない → 相談に返事がつかない | 創業者自身が最初の回答者に。経験者10人を事前リクルーティング |
| 回答者による勧誘行為 | ロール別監視、「個人的見解」チェック、通報体制 |
| 詐欺的勧誘業者の流入 | 重点監視・招待制スタート |
| 非弁行為認定 | 回答者の「個人的見解」チェック、断定表現の自動検知 |
| 危機対応の失敗 | 困りごと選択の `triggers_crisis` フラグ + キーワード検知 |
